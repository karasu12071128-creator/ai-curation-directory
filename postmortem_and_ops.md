# Postmortem & 運用手順

## 1. Postmortem: 「0 items」問題の原因と対応

### 1.1. 問題の概要
サイトのResources一覧ページ（list.html）において、データがロードされず「0 items」と表示され、利用者がリソースを閲覧できない状態になった。

### 1.2. 原因の特定
問題は**フロントエンドのデータ取得ロジック**に起因していました。

元のコードでは、データ取得のフォールバックパスが不十分であり、特にGitHub Pagesのデプロイ環境でデータファイル（`resources.json`）のパス解決に失敗した場合、データが空の配列（`[]`）として処理されていました。

```javascript
// 修正前のデータ取得ロジック（list.html内）
data = await tryFetch('{{ "/resources.json" | relative_url }}')
    || await tryFetch('{{ "/data/resources.json" | relative_url }}')
    || []; // パス解決失敗時に空配列がセットされていた
```

データファイル自体（`docs/resources.json`）はGitHubリポジトリ内に存在し、内容も空ではありませんでした。

### 1.3. 対応と暫定復旧
1.  **データ取得ロジックの強化:** `list.html`内のデータ取得ロジックに、ローカル開発環境や異なるデプロイパスに対応するためのフォールバックパス（`./resources.json`、`../resources.json`）を追加しました。これにより、データが正常にロードされる状態に復旧しました。
2.  **UI/UXの改善:**
    *   **最終更新日時と件数**の表示を追加。
    *   **0件時のエラーメッセージ**と**「フィルターをリセット」ボタン**を追加し、利用者が問題を解決できるようにしました。

### 1.4. 再発防止策（実装済み）
1.  **ビルド前テスト（items > 0 のアサート）:** GitHub Actionsの`daily.yml`に、データ生成後の`resources.json`のアイテム数が0件でないことを確認するステップを追加しました。0件の場合はワークフローを失敗させ、デプロイをブロックします。
2.  **フロントエンドのエラーハンドリング:** データ取得失敗時でも、UIが利用者に適切なフィードバック（リセットボタンなど）を提供するように改善しました。

## 2. 運用手順

### 2.1. 手動リカバリ手順（データ更新に失敗した場合）
データ更新ジョブが失敗した場合や、公開されているデータが0件になった場合は、以下の手順で手動リカバリを実行します。

1.  **GitHubリポジトリの「Actions」タブに移動します。**
2.  **左側のワークフロー一覧から「daily-refresh」を選択します。**
3.  **「Run workflow」ボタンをクリックし、表示されるダイアログで再度「Run workflow」をクリックして、手動でジョブを再実行します。**

ジョブが正常に完了すると、`resources.json`と`resources.csv`が再生成され、サイトにデータが反映されます。

### 2.2. 監視の確認方法
データ更新ジョブの成否は、GitHub Actionsの実行結果で確認できます。

*   **ジョブ失敗時の通知:** ジョブが失敗した場合、または生成されたデータが0件だった場合は、GitHub Actionsが自動的にエラーを検出し、リポジトリの管理者に通知メールが送信されます。
*   **週次レビュー:** 毎週金曜日の12:00 JSTにPlausibleダッシュボードを確認し、KPI（特に`Open` CVRや`404`件数）に異常がないかを確認してください。
